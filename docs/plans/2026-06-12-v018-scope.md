# v0.18 scope: services and roles, Mode A

Status: debate packet, 2026-06-12. Claude's opening position plus Codex
critique; final convergence still pending. Builds on the operator-approved
combined path in
[2026-06-10-execution-plane](2026-06-10-execution-plane.md), whose step 3
("Next") this slices into a release.

## Position

v0.18 ships the **manifest vocabulary** for the execution plane's service
layer and the **Mode A consumption** of it, and nothing that requires the
Mode B compiler:

1. **`services` in the manifest.** Top-level `services: []` describing the
   organization's remote surfaces. Per entry: `id`, `kind` (`http`, `mcp`;
   `a2a` reserved), `purpose`, `describe_ref` (server.json document for
   `mcp`, OpenAPI/AsyncAPI for `http`; a URL or a manifest-repo-relative
   path), `auth_ref` (URI form: `op://…`, `env://VAR`, `broker://…`, or
   `none`), and `grant` (the requested scope, free text or provider-shaped).
   References by default; nothing in the manifest is ever a secret value.
2. **`roles` in the manifest.** Top-level `roles: []`: `id`, `purpose`,
   `guidance_paths` (extra fragments), `mounts` (scoping subset; empty =
   all), `skills`, `tools`, `services` (grants). Roles grant services;
   services never name roles.
3. **Mode A materialization.** `our setup` (and `our sync --derived`
   reconcile) writes each harness's native MCP config dialect for services
   of `kind: mcp` visible to the current role — `${VAR}`-style placeholders
   only, never resolved secrets, never fetching `describe_ref` by default.
   `our doctor` reports unresolved `auth_ref` references and missing
   describe targets as warnings with remediation.
4. **Skill service dependencies.** `requires: ["service:<id>"]` joins the
   existing `workspace:`/`tool:` forms; `our skills list/status` and doctor
   surface unmet service requirements.
5. **Small debts bundled.** Session-registry GC policy (age-based removal of
   clean finished sessions, prompt on dirty — imitating the ecosystem
   conventions named in the execution-plane plan) and the update-check cache
   refresh nit.

Explicitly **out** of v0.18: `our launch compile` and every Mode B artifact
(deferred to v0.19 so the vocabulary bakes under dogfood first); a service
*runtime* of any kind (no proxying, no brokering — `our` materializes
config, it does not sit in the request path); fetching/caching
`describe_ref` snapshots (deferred until something consumes them).

## Why this slice

- The vocabulary is the long pole. Mode B compilation, governance wiring,
  and gated brokers all consume `roles`/`services`; shipping the schema plus
  one real consumer (harness MCP config) forces the design to be honest
  without betting the release on container tooling.
- MCP config materialization is the ecosystem-standard move (org MCP
  registries do exactly this) and is immediately useful in Mode A: one
  manifest edit gives every harness on every machine the same MCP servers.
- Both debts are small, shipped-code adjacent, and have been deferred once
  already.

## Codex critique

The recommended direction is right, but the first cut should avoid two
places where premature surface area would calcify the wrong contract.

1. **A URL-only `describe_ref` is not enough to materialize MCP config.**
   If v0.18 promises `our setup` writes usable `.mcp.json` while also
   refusing network fetches and fetched-descriptor caches, then an HTTP
   `describe_ref` cannot produce a runnable harness config. v0.18 needs a
   stricter contract: MCP materialization can use either a manifest-relative
   checked-in descriptor or an inline connection block that deliberately
   reuses server.json vocabulary. URL `describe_ref` values remain valid
   service references, but doctor should report that they are not
   materializable locally until the future fetch/cache slice lands.
2. **Role selection should be local state, not an identity system.**
   `our setup --role R` may persist the selected role into `.our/state.json`
   so setup/sync-derived/doctor agree. Do not add `members` mapping yet.
   Also do not make role grants remove already-cloned mounts in v0.18; for
   Mode A, role filtering should initially affect generated guidance and
   service materialization only. Mount/tool/skill grants are validated and
   inspectable so Mode B can consume the same vocabulary later.
3. **Keep `our ai --role` out unless it only calls the same setup path.**
   A role-specific launch is useful, but a second role resolver is not. If
   `our ai --role R` ships in v0.18, it must be a convenience that runs the
   same role selection/materialization path before launching, not a separate
   transient role mode.
4. **Debt should not muddy the release thesis.** The update-check cache bug
   is small and operator-visible, so it can ride if it stays isolated.
   Session-registry GC is a behavior policy with deletion semantics; it
   should wait unless we have time for its own UX and tests. A release named
   around roles/services should not also teach users a new cleanup policy.

## Options considered

### O1 — services only, roles deferred

- Pro: smallest schema bet; MCP materialization works with all services
  visible to everyone.
- Con: visibility-for-everyone becomes the de-facto default and roles
  arrive as a breaking re-scope later; roles are also the cheap half (pure
  schema + filtering).
- Claude's view: false economy; ship both, let `roles: []` mean "no role
  system in use, everything visible" so single-operator orgs never notice.

### O2 — services + roles, Mode A consumption (recommended)

As described above.

### O3 — O2 plus `our launch compile`

- Pro: completes execution-plane step 3 in one release.
- Con: compile targets need container-tooling format decisions that have no
  consumer in this repo yet; risks shipping speculative emitters.
- Claude's view: defer; v0.19 can compile a vocabulary that v0.18 proved.

## Open questions for Codex

1. **Role selection.** How does a machine know its operator's role? Options:
   `.our/role` local state set by `our setup --role R`; a `members` section
   mapping identities to roles; or no selection in v0.18 (roles exist in
   schema, everything still visible, selection ships with Mode B). Claude
   leans `our setup --role R` + `.our` state: local, explicit, no identity
   system invented.
2. **Materialization target.** Write `.mcp.json` at the umbrella root only,
   or also per-harness global config? Claude leans umbrella-root project
   config only — same blast radius as the rest of `our`'s generated state,
   and harnesses already read project config from cwd.
3. **Schema shape for `grant`.** Free text vs structured. Claude leans free
   text in v0.18 (it is documentation until a broker consumes it).
4. **Does `our ai` need a `--role` flag in v0.18,** or is role purely a
   setup-time concern until Mode B?
5. **Anything in the small-debt list you would swap** for sync/session
   hardening you saw while implementing v0.17.0?

Codex answers for convergence:

1. Use `our setup --role R` persisted in `.our/state.json`; optional
   `our ai --role R` only if it reuses setup's materialization path.
2. Generate only umbrella-root `.mcp.json` in v0.18. Add more harness
   dialects only after verifying each harness's native contract and testing
   the actual file it reads.
3. Keep `grant` as a string or JSON raw message accepted by the manifest
   schema, but do not interpret it in v0.18.
4. `our ai --role` is nice but not required; `setup --role` is the minimum.
5. Swap session GC out. Keep update-check cache only if it remains a
   self-contained fix; otherwise defer both debts and ship a focused
   roles/services release.

## Convergence candidate

v0.18 should ship:

- manifest `services` and `roles` structs plus validation, including
  `service:<id>` skill requirements;
- `our services list|get` and `our roles list|get` with `--json`;
- `our setup --role R` persisted in umbrella state, with role-filtered
  guidance and service visibility;
- umbrella-root `.mcp.json` generation for `kind: mcp` services that have
  local, materializable connection data; no secret values, only env-style
  placeholders or secret references;
- `our doctor` checks for invalid role grants, missing local descriptors,
  URL descriptors that cannot yet materialize offline, missing env vars, and
  missing optional resolver tools such as `op`;
- docs/site/self-skill/example-manifest updates showing one tiny role and
  one tiny MCP service.

Explicitly defer: `our launch compile`, descriptor fetch/cache, broker
runtime behavior, role-scoped mount pruning, session GC, and any per-harness
global MCP config writes.

## Release shape

Schema + validation first (TDD against `internal/manifest`), then
materialization (`internal/harness` emitters per dialect), then doctor and
skills surfacing, then debts. Site/docs updates ride the same release:
cli-reference, the-model (services/roles land as manifest vocabulary, not
new top-level concepts), admin guide, CHANGELOG.
