# ADR 0001: Launch-scoped skill composition

Status: **accepted**. Drafted 2026-06-14 after multi-agent design review.
Pre-1.0, no userbase: back-compat is not a constraint. Implementation is
tracked in
[`2026-06-14-launch-scoped-skill-composition`](../plans/2026-06-14-launch-scoped-skill-composition.md);
this ADR fixes the direction only.

## Discussion summary

- Management tiers need different skill loadouts. Executive finance/payroll
  workflows should not be present in a software-manager launch, and the number
  of organization skills will grow enough that "install everything everywhere"
  is the wrong default.
- `our` provisions local derived state; it is not an authorization system. If a
  sensitive skill lives in a manifest repo, anyone who can clone that manifest
  can read it. Confidentiality remains a Git/service ACL boundary, not a role or
  profile flag.
- Harnesses increasingly support workspace/project skills, but their discovery
  semantics differ. Some look for `.agents/skills`, some also support
  harness-specific directories, and none exposes a portable "load these skills"
  CLI flag. Therefore `our` must compose the skill set deterministically instead
  of delegating composition to harness tree-walk behavior.
- User-level reconciliation was rejected. The operator regularly runs multiple
  harnesses in different contexts on the same machine, so rewriting shared
  `~/.../skills` directories per launch would create collisions.
- A second invocation-root lifecycle was rejected. Work sessions already own the
  durable local work surface. Skill exposure is a disposable launch profile that
  should be regenerated into the existing launch root, not a parallel
  `.our/invocations/<id>` tree.

## Problem

Today the supported harnesses install managed skills at user-home level:
Claude Code, Codex, and OpenCode use filesystem skill directories under their
user config dirs, while the current code treats Gemini as a CLI-managed special
case. See `harness.SkillTargetPath` in `internal/harness/harness.go` and the
installer in `internal/skills/skills.go`.

That set is flat and global: it is not scoped by role, repo, mount, or launch
context. `our ai` already chooses a launch root by context
(`launchTargetDir` in `internal/cli/launch.go`: base umbrella, `--repo <id>`,
or active/new work session) and ensures the bundled self-skill before exec, but
it does not compose a contextual skill set.

Three forces make the flat-global model wrong going forward:

1. **Least exposure.** Different roles, repos, and work sessions need different
   workflow affordances.
2. **Growth and context caps.** The skill count grows with the organization.
   Harnesses have finite skill/context budgets, so an uncurated global set
   makes every launch noisier and less reliable.
3. **Concurrency.** Multiple harnesses may run at once in different contexts.
   Shared mutable user-level skill directories make those launches fight.

## Decision

**`our ai` is the skill-profile composer. It computes the effective skill set
for the launch and materializes exactly that set as disposable derived state in
the launch root. Organization skills are not installed at user level for managed
launches.**

Two axes stay separate:

- **Session**: the durable assembled work surface and lifecycle. Today sessions
  assemble content-mount worktrees plus scratch under `work/<id>`;
  repo-inclusive sessions remain a follow-up. Sessions answer "what files and
  mounts are in the work surface?"
- **Profile**: the skills, and their required tools/services, exposed for one
  launch. It is regenerated by `our ai`, has no independent lifecycle, and is
  written into the selected launch root.

The selected launch root may be the umbrella root, an active/new session root,
or a direct `repos/<id>` checkout. Direct repo checkouts need an explicit
cleanliness policy before implementation; see consequences below.

## Mechanism

1. **Minimal global footprint.** The bundled `our` self-skill may remain
   broadly available during migration for harnesses that still require it, but
   the target state is launch-root materialization for all managed skills,
   including the self-skill where the harness supports it.
2. **Profile inputs, not just skill-owned scope tags.** The schema should let
   roles, named profiles, repos, products, mounts, and explicit CLI selectors
   choose skills. A skill may also carry default-exposure metadata such as
   "always", "targeted", or "manual" (final field names are plan work). Avoid a
   model where every skill has to hard-code every role that might use it;
   `Role.Skills` already exists and should remain a first-class selector.
3. **Closure over requirements.** If an in-scope skill requires tools or
   services, the profile composer must either include them or fail with a
   precise error. `our compile` already validates role-selected skill closure in
   `internal/launchplan/launchplan.go`; launch composition should use the same
   principle.
4. **Selector on `our ai`.** The portable argument is on `our`, not the harness:
   `our ai --skills ...`, `our ai --profile ...`, plus defaults inferred from
   the target root. Harness flags such as `--cd`, `--add-dir`, or Claude's
   plugin flags are not a cross-harness skill API.
5. **One generated skill directory per launch root.** `our ai` materializes a
   managed `.agents/skills/` in the launch root and wipes/regenerates only
   entries it owns. Entries should link to canonical skill sources where safe,
   with a copy fallback for harnesses or filesystems that do not handle symlinks
   correctly. Harness-specific mirrors, for example `.claude/skills`, are
   generated only when a harness has a proven launch-root skill seam and does
   not read `.agents/skills`. A harness without a proven launch-root seam is a
   compatibility exception, not a mirror target: keep its existing global skill
   path and do not claim per-launch selectors work for it.
6. **No scattered discovery hacks.** `our` should not auto-symlink arbitrary
   skills found in subfolders. Repo-local committed skills may exist, but they
   should be either declared as sources or left to the harness; `our` must not
   clobber a non-managed `.agents/skills` tree.

## Profile selection

Defaults are target-driven and overrideable:

- Base umbrella launch: broad organization default.
- `our ai --repo <id>`: self-skill + repo-associated skills + organization
  skills marked for all launches. Broad organization workflows do not load
  unless requested.
- Session launch: self-skill + skills associated with the session's assembled
  mounts, and with the target repo once repo-inclusive sessions exist.
- Explicit override: `--skills all`, `--skills none` (self-skill only),
  `--skills a,b,c`, or a named `--profile <name>`.

Example product shape:

```sh
our ai agy                                      # broad umbrella default
our ai --repo orders-api agy                    # self + repo + always-org
our ai --repo orders-api --skills all agy       # widen workflows
our ai --profile finance-reconcile claude       # named profile
our ai --skills meetings,support codex          # explicit subset
```

## Skill scope is not context scope

Skills are workflow affordances. They are not the data-access mechanism.

Access to organization context comes from the data plane: synced mounts,
services, credentials, catalog metadata, and the always-available `our` CLI
surface that can locate the umbrella from a nested launch root. A repo-scoped
launch can therefore have a narrow skill profile while still using authorized
company context through the data plane.

This keeps the founding motivation intact: an agent can work on a repo with
company context without preloading every company workflow skill. Workflow
breadth is selected by the skill profile (`--skills all`, named profiles, or
target defaults). Data breadth is selected by which mounts/services/session
roots are available and authorized.

The same distinction is also why profiles are not a security boundary. If a
manager must not read payroll material, the payroll skill and backing data must
live behind manifest, repo, or service ACLs that the manager cannot access.

## Alternatives rejected

- **User-level reconcile.** Rewriting `~/.claude/skills`, `~/.codex/skills`, or
  similar user directories per launch is simple but breaks concurrent harnesses.
  It remains acceptable for personal/manual installs, not managed organization
  launch profiles.
- **A second `.our/invocations/<id>` lifecycle.** This duplicates sessions.
  Profiles are derived state attached to an existing launch root.
- **Delegating composition to harness tree traversal.** Discovery rules differ
  across harnesses and do not provide a reliable portable merge of umbrella,
  repo, role, and session intent.
- **Passing skills as a harness CLI argument.** No common flag exists. The
  selector belongs on `our ai`.
- **Arbitrary subfolder symlink generation.** It hides policy in filesystem
  layout and creates messes that are hard to inspect or clean. Skill selection
  should be declared and materialized into one managed directory.
- **Roles as confidentiality.** Roles and profiles select loadout. They do not
  prevent someone from reading files in a manifest they can clone.

## Consequences

- `our setup` should stop treating organization skills as user-global derived
  state for managed launches. Existing user-level commands become migration and
  manual/personal-skill surfaces.
- `our ai` gains profile resolution, requirement closure, and launch-root
  materialization at the existing `launchTargetDir`/self-skill seam.
- The manifest needs additive fields for named profiles and for associating
  skills with repos, products, mounts, roles, and default exposure. The plan
  should decide exact field names.
- The harness matrix needs updating as Gemini is retired and Antigravity (`agy`)
  becomes the successor adapter. The architecture should center on
  `.agents/skills` plus per-harness mirrors, not on either Gemini's old install
  path or a single harness's project-skill behavior.
- Direct `--repo` launches are the main cleanliness risk. Writing generated
  `.agents/skills` into a real checkout can dirty or conflict with the repo.
  Preferred path: route repo work through session roots once repo-inclusive
  sessions exist. Until then, direct repo launches must either use a local
  exclude/cleanup policy and refuse to clobber non-managed trees, or remain a
  narrower admin/inspection surface.
- Tests need to cover profile selection, role/profile closure, managed-directory
  cleanup, non-managed `.agents/skills` refusal, concurrent launch roots, and
  harness mirror generation.
- Confidentiality for sensitive tiers remains orthogonal: restricted
  manifests/repos/services with real ACLs.

## Follow-ups

1. Write the implementation plan under `docs/plans/`, including schema fields,
   selector syntax, materialization layout, cleanup/doctor behavior, repo
   checkout policy, and the Gemini-to-Antigravity adapter transition.
2. Decide the canonical on-disk source model for skills so launch-root links or
   copies are portable and inspectable.
3. Extend sessions or launch targeting so repo-associated work can use a clean
   generated root instead of writing derived state into base repo checkouts.
